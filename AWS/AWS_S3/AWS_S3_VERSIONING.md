# AWS S3 Versioning

## What is Versioning?

Amazon S3 Versioning is a feature that keeps **multiple versions of the same object** in a bucket. Instead of overwriting or permanently deleting an object, S3 assigns a unique **Version ID** to every upload.

---

## Real-Life Example

Imagine editing `resume.pdf` multiple times.

Without Versioning:

```
resume.pdf
↓
Overwritten
```

With Versioning:

```
resume.pdf
├── Version 1 (Original)
├── Version 2 (Added Skills)
├── Version 3 (Updated Experience)
└── Version 4 (Latest)
```

You can restore any previous version if needed.

---

## Is Versioning a Backup?

**No.**

Versioning is a **data protection feature**, not a backup solution.

Versioning stores multiple versions **within the same bucket**. If the entire bucket is deleted, all versions are lost.

A true backup stores data in another independent location, such as:

- Another S3 bucket
- Cross-Region Replication (CRR)
- AWS Backup
- External backup storage

---

## Why Use Versioning?

- Protects against accidental deletion
- Protects against accidental overwrite
- Maintains object history
- Enables rollback to previous versions
- Supports ransomware recovery when combined with Object Lock

---

## How Versioning Works

Uploading the same object multiple times:

```
notes.txt

Version 3 (Latest)
Version 2
Version 1
```

The object key stays the same, but each upload receives a unique **Version ID**.

---

## Version IDs

Example:

```
notes.txt

Version ID: 3HL4kqtJlcpXrof3...
Version ID: d9Ks82Jsks93...
Version ID: AAA129382...
```

Every upload receives a unique Version ID generated by AWS.

---

## Delete Marker

Deleting an object in a versioned bucket does **not** immediately remove all versions.

Instead, S3 creates a **Delete Marker**.

```
Delete Marker
↓
Version 3
↓
Version 2
↓
Version 1
```

Removing the Delete Marker restores access to the latest object version.

---

## Versioning States

### 1. Unversioned (Default)

Objects are overwritten when uploaded with the same key.

### 2. Versioning Enabled

Every upload creates a new version.

### 3. Versioning Suspended

Existing versions remain intact, but new uploads no longer create additional versions.

---

## Can Versioning Be Disabled?

No.

Once enabled, Versioning **cannot be disabled**. It can only be **suspended**.

---

## Storage Cost

Every version consumes storage.

Example:

```
movie.mp4 = 5 GB

Version 1 = 5 GB
Version 2 = 5 GB
Version 3 = 5 GB
```

Total billed storage:

```
15 GB
```

---

## Lifecycle Integration

Lifecycle Rules help control costs by:

- Deleting old versions
- Transitioning older versions to Glacier
- Expiring previous versions automatically

---

## Replication

Versioning must be enabled for S3 Replication.

All object versions are replicated to the destination bucket.

---

## Object Lock Integration

When combined with Object Lock:

- Object versions become immutable
- Even administrators cannot delete protected versions until the retention period expires

---

## Advantages

- Protects against accidental overwrite
- Protects against accidental deletion
- Supports rollback
- Maintains version history
- Integrates with Replication, Lifecycle, and Object Lock

---

## Disadvantages

- Higher storage costs
- More versions to manage
- Not a replacement for backups

---

## Interview Questions

### Is Versioning a backup?

No. It protects against accidental deletion and overwrite but stores all versions in the same bucket.

### Can Versioning be disabled?

No. It can only be suspended.

### What happens when an object is deleted?

A Delete Marker is created, while previous versions remain available.

### Does every upload create a new object?

No. The object key remains the same, but each upload receives a new Version ID.

### Does Versioning increase storage costs?

Yes. Every version is billed separately.

### Can Lifecycle Rules delete old versions?

Yes. Lifecycle Rules can automatically remove or archive previous versions.

## Memory Trick

**Versioning = Undo History**

Like Google Docs or Microsoft Word, every save creates another version that you can restore.

**Backup = Copy stored somewhere else**

If the original location is lost, the backup still exists.
